# Provenance Guard

A backend API for attributing creative text content — detecting whether a submitted piece
of writing was generated by an AI or authored by a human, scoring confidence, surfacing
a transparency label, and handling creator appeals.

## Architecture Overview

The path a submission takes from input to transparency label:

```
POST /submit
    │
    ▼
[Rate Limiter]          → 429 if limit exceeded
    │
    ▼
[Input Validator]       → 400 if text or creator_id missing
    │
    ├── Signal 1: LLM Classifier (Groq)
    │       Sends text to llama-3.3-70b-versatile
    │       Returns llm_score: float 0.0–1.0
    │
    └── Signal 2: Stylometric Heuristics (pure Python)
            Computes sentence-length variance, type-token ratio,
            punctuation density
            Returns style_score: float 0.0–1.0
    │
    ▼
[Confidence Scorer]     → confidence = 0.60 × llm_score + 0.40 × style_score
    │
    ▼
[Label Generator]       → one of three transparency labels
    │
    ▼
[Audit Logger]          → structured row written to SQLite
    │
    ▼
[JSON Response]         → content_id, attribution, confidence, label_text, ...
```

**Appeal flow:** `POST /appeal` → look up content_id → update status to `under_review`
→ insert appeal row in SQLite → return confirmation.

## Detection Signals

### Signal 1 — LLM Classifier (Groq, `llama-3.3-70b-versatile`)

**What it measures:** The holistic semantic "AI-ness" of text. The model has
internalized patterns common in AI output: hedged phrasing ("it is important to
note"), over-structured paragraphs, suspiciously even register, generic examples,
absence of personal voice or emotional texture.

**Output:** A float `0.0–1.0` where `1.0` means "strongly AI-generated." The model
returns structured JSON: `{"ai_probability": 0.83, "reasoning": "..."}`.

**Why chosen:** It captures contextual and semantic properties that statistical
heuristics cannot — a sentence can have normal length variance but still read exactly
like GPT output. The reasoning field also provides human-readable transparency.

**Blind spot:** The LLM may penalize formal human writing. A carefully revised academic
essay, or writing by a non-native English speaker aiming for correctness, can
pattern-match to AI output. This is the primary source of false positives.

---

### Signal 2 — Stylometric Heuristics (pure Python)

**What it measures:** Statistical uniformity across three independent sub-metrics:

| Sub-metric | What it captures | AI pattern |
|-----------|-----------------|------------|
| **Sentence length variance** | Std dev of word count per sentence | AI sentences cluster around a narrow range |
| **Type-token ratio (TTR)** | Unique words / total words | AI repeats vocabulary at a more regular rate |
| **Punctuation density** | Punct chars / total chars | AI under-uses dashes, ellipses, exclamation marks |

**Sub-metric weights:** `0.40 × variance + 0.35 × TTR + 0.25 × punctuation`

**Output:** A float `0.0–1.0` where `1.0` means high statistical uniformity (AI-like).

**Why chosen:** Completely independent of the LLM. Computable in pure Python with no
API calls. Even when both signals agree, they arrive at their conclusion through
completely different evidence — which makes the combination more reliable.

**Blind spot:** Short texts (< 100 words) have naturally high TTR (most words are
unique), which compresses all short texts toward mid-range style scores. Stylometrics
work best on longer passages (500+ words).

## Confidence Scoring

### Combining signals

```
confidence = 0.60 × llm_score + 0.40 × style_score
```

**Why 60/40?** The LLM captures semantic nuance that statistics cannot. But giving
it 100% weight would make the system a black box — the stylometric signal adds
interpretability and serves as a check when the LLM is overconfident.

### Score → attribution mapping

| Score range | Attribution code | Rationale |
|------------|-----------------|-----------|
| 0.00 – 0.29 | `likely_human` | Strong evidence of human authorship |
| 0.30 – 0.69 | `uncertain` | Mixed signals; system cannot confidently decide |
| 0.70 – 1.00 | `likely_ai` | Strong evidence of AI generation |

**Why not 0.50?** The spec hints at a key asymmetry: a false positive (labeling a
human's work as AI) is worse than a false negative. We set the AI threshold at 0.70
(not 0.50) to give human creators the benefit of the doubt when signals are mixed.

### Example submissions with noticeably different scores

**High-confidence AI (confidence: 0.725)**
```
Input: "Artificial intelligence represents a transformative paradigm shift in modern
society. It is important to note that while the benefits of AI are numerous, it is
equally essential to consider the ethical implications. Furthermore, stakeholders
across various sectors must collaborate to ensure responsible deployment."

→ llm_score:   0.85   (LLM spotted hedging language and even register)
→ style_score: 0.537  (moderate uniformity in short text)
→ confidence:  0.725  → likely_ai
```

**High-confidence human (confidence: 0.253)**
```
Input: "ok so i finally tried that new ramen place downtown and honestly?
underwhelming. the broth was fine but they put WAY too much sodium in it and
i was thirsty for like three hours after..."

→ llm_score:   0.05   (casual phrasing, typos, personal voice)
→ style_score: 0.557  (short text, mid-range stylometrics)
→ confidence:  0.253  → likely_human
```

### Validation approach

Both signals were tested independently on 4 deliberately chosen inputs (clearly AI,
clearly human, two borderline cases) before integration. The clearly AI and clearly
human samples produce markedly different LLM scores (0.85 vs 0.05), confirming the
LLM signal is the primary differentiator. The stylometric signal adds signal on longer
text and serves as a consistency check.

## Transparency Labels

All three variants, exactly as returned by the API:

### Variant 1 — High-Confidence AI (`confidence ≥ 0.70`)

```
⚠️  AI-Assisted Content Detected
Our analysis found strong indicators that this content was generated or heavily
assisted by AI tools (confidence: 72%). This label helps readers understand
the content's origins.

If this is your original work and you believe this is incorrect, you have the
right to appeal this classification. Appeals are reviewed by our team.
```

### Variant 2 — Uncertain (`0.30 ≤ confidence < 0.70`)

```
ℹ️  Content Origin: Uncertain
Our system detected mixed signals and cannot confidently determine whether this
content is human-written or AI-assisted (confidence: 37%). No action has been
taken on your content.

If you'd like to clarify the origin of this work, you may submit an explanation
via our appeals process.
```

### Variant 3 — High-Confidence Human (`confidence < 0.30`)

```
✓  Original Human Authorship
Our analysis found strong indicators of original human authorship
(75% human confidence). No action required.
```

> Note: The percentage in the label is the AI confidence for Variants 1 & 2, and the
> human confidence (1 − AI confidence) for Variant 3, so the number always reads
> intuitively to a non-technical reader.

| Window | Limit | Reasoning |
|--------|-------|-----------|
| Per minute | 10 requests | A creator submitting their own work will not exceed this. A script flooding the system will be blocked within seconds. |
| Per day | 100 requests | Generous enough for power users who submit multiple drafts. A bot making hundreds of daily calls is blocked. |

## Known Limitations

### 1. Formal human writing is the primary false-positive source

A well-educated writer who uses structured arguments, hedged language ("research
suggests," "it is worth noting"), and avoids colloquialisms will score high on both
signals. The LLM has seen vast amounts of AI-generated text that looks exactly like
formal academic writing. This is the most likely real-world misclassification.

**Why it's hard to fix:** The LLM's training data conflates "formal" with "AI-like."
Fixing this would require fine-tuning on labeled formal human writing — a problem that
requires a dataset the current system doesn't have.

**Mitigation in this system:** The appeals workflow is explicitly designed for this
case. The "Uncertain" label band (0.30–0.69) is wide, and the AI threshold is 0.70
(not 0.50), giving formal human writers extra margin before being flagged.

### 2. Short text (< 100 words) compresses stylometric scores

The type-token ratio sub-metric loses discriminative power on short texts because
nearly every word in a short passage is unique regardless of whether it's AI or human.
This makes the `style_score` cluster around 0.5 for short inputs and the LLM signal
dominates entirely — which works, but eliminates the benefit of the two-signal design.

## Spec Reflection

**One way the spec helped:**  
The spec's insistence on writing out the three exact label variants *before* implementation was valuable. It forced a concrete decision about what "uncertain" looks like to a non-technical reader — specifically that it should NOT flag the creator ("No action has been taken on your content") and should explicitly offer the appeals path. Without writing this out first, implementation would likely have produced a vague "we're not sure" message that feels punitive rather than neutral.

**One way implementation diverged from the spec:**  
The spec called for a 60/40 LLM/stylometric weighting, but testing revealed that the stylometric signal produces almost no discrimination on short texts (all scores cluster 0.52–0.57). In practice, the system behaves more like 80/20 for short inputs, because the style_score adds little signal. A more honest implementation would detect short texts and either drop the stylometric weight or widen the "uncertain" band when `len(text.split()) < 100`. This is documented as a known limitation rather than patched mid-implementation.

## AI Usage

### Instance 1 — Database module design

**What I directed the AI to do:** Generate a SQLite module with two tables
(`submissions` and `appeals`) that supports the submission + appeal flows, with a
`get_log()` function that merges both tables and returns them sorted by timestamp.

**What it produced:** A working module using `sqlite3.Row` for dict-like access, but
it used `LEFT JOIN` in `get_log()` which duplicated submission rows when multiple
appeals existed for the same content_id.

**What I revised:** Rewrote `get_log()` to query the two tables separately, merge
in Python, and sort — avoiding the JOIN duplication problem entirely. This also made
the `type` field (`"submission"` vs `"appeal"`) explicit per entry, which is more
useful for graders reading the log.

### Instance 2 — Stylometric heuristics normalization

**What I directed the AI to do:** Generate a `compute_stylometric_score()` function
with sentence-length variance, type-token ratio, and punctuation density sub-metrics,
each normalized to 0.0–1.0.

**What it produced:** A correct structure but with normalization constants that didn't
account for short texts. The type-token ratio normalization assumed TTR would range
0.3–0.9, but for short texts (< 100 words) TTR is nearly always 0.8+, so all short
texts scored near 0 on TTR (high vocabulary diversity → looks human).

**What I revised:** Kept the normalization constants but added a comment explaining
why the TTR signal compresses on short text — and documented this as a known
limitation rather than trying to add length-dependent normalization (which would be
premature complexity).
